## Why this is split out

The functional core of incremental materialized views (refresh policy, eligibility
gate, manager + delta subscription, per-binding delete-then-upsert apply, cost
fallback, drop detach, schema-change invalidation) **landed** under
`materialized-view-incremental-refresh` (now in review). This ticket is the
remaining Phase-D change-scope projection, which the parent ticket explicitly
called out as the clean split seam. It is orthogonal to maintenance correctness:
it only affects what `Statement.getChangeScope()` / `Database.watch` report for a
query that reads an incremental MV.

## Problem

A `select` from an MV resolves to a `TableReferenceNode` over the MV's **backing
table** (`sqlite_mv_<name>`). `analyzeChangeScope` therefore currently reports the
*backing table* as the watched table. But an `on-commit-incremental` MV's backing
table is never written by user DML — it is maintained at COMMIT from its sources.
So a `Database.watch` on such an MV would never fire (nothing writes the backing
table through the user change log). It should instead fire when a **source**
changes.

## Design

- Cache, on `MaterializedViewSchema`, the **source-union change-scope** for an
  `on-commit-incremental` MV — computed once at create time (and on
  re-registration). v1 may use the conservative union of a `full` watch per
  source table (`unionScopes`, `change-scope.ts:681`); a precise per-source
  row/group scope (mirroring the maintenance bindings the manager already derives)
  is a refinement. Field e.g. `sourceScope?: ChangeScope`.
- Teach `analyzeChangeScope` (`change-scope.ts:203`) to detect "this
  `TableReferenceNode` is an `on-commit-incremental` MV backing table" and replace
  its `TableWatch` with the cached source-union scope. Detection needs schema
  access, which `analyzeChangeScope` does not currently have — thread an optional
  resolver through its options (the sole caller is `Statement.getChangeScope`,
  `core/statement.ts:611`, which has `this.db`). Detect via the schema manager
  (backing-table name ↔ MV lookup; `backingTableNameFor` / the `sqlite_mv_`
  prefix). Add a `SchemaManager` reverse lookup (`getMaterializedViewByBackingTable`).
- `manual` MVs keep reporting just the backing table (their cadence is `refresh`).
- Cross-reference: `docs/optimizer.md` § Binding-aware Delta and
  `docs/change-scope.md` should note the MV-reference projection (the
  `docs/materialized-views.md` § Incremental refresh "Limitations" bullet already
  forward-references this slug — flip it to "delivered" on landing).

## Tests

Extend `test/logic/52-materialized-views-incremental.sqllogic` (or a `test/`
unit spec): `getChangeScope()` on a query reading an incremental MV reports the
**source** table(s), not `sqlite_mv_*`; a `Database.watch` on the MV fires on a
source mutation. A manual-refresh MV still reports the backing table.
