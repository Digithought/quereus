## Goal

A post-optimization rule that walks the `SetOperationNode(op='unionAll')` chain rooted at any relational node, flattens it into a single N-ary `AsyncGatherNode(unionAll)` when every child clears two gates:

1. `physical.expectedLatencyMs ≥ tuning.parallel.gatherThresholdMs` for at least the *max* child (i.e. some child is genuinely high-latency).
2. Every child is `physical.concurrencySafe === true`.

The cost gate ensures local-only plans (`expectedLatencyMs === 0` on every child) never trigger the rewrite. The fanout rule (`4.5`) established the same skip-when-zero invariant — the gather rule reuses both the field and the convention.

This rule depends on `5-parallel-async-gather-node` (the node + emitter) and on `4.5-parallel-fanout-lookup-join-rule` (which lands the `expectedLatencyMs` / `concurrencySafe` fields on `PhysicalProperties`, their propagation, and the `parallel` section in `OptimizerTuning`).

## Architecture

### New tuning knob

Extend the `parallel` block (added by `4.5`) with:

```ts
readonly parallel: {
  // existing from 4.5:
  readonly minBranches: number;
  readonly branchSetupCost: number;
  readonly concurrency: number;
  // new in this ticket:
  /**
   * The slowest child of a unionAll chain must have at least this expected
   * first-row latency (in milliseconds) for the gather rule to fire. Set
   * high enough that local-only plans never trigger — defaults to a value
   * that any synthetic test fixture must explicitly exceed.
   */
  readonly gatherThresholdMs: number;
};
```

Add the default to `DEFAULT_TUNING`. Suggested value: **25 ms** (matches the synthetic high-latency vtab fixture used by the fanout tests, so the gather rule fires under the same fixture). Rationale documented inline.

### Recognition rule

`packages/quereus/src/planner/rules/parallel/rule-async-gather-union-all.ts`:

```
ruleAsyncGatherUnionAll(node, context):
  if node.nodeType !== PlanNodeType.SetOperation: return null
  if (node as SetOperationNode).op !== 'unionAll': return null

  // Flatten the left-deep chain. SQL `A UNION ALL B UNION ALL C` parses to
  // SetOp(SetOp(A, B), C) — left-associative. We accept any tree shape so
  // long as every internal node is a unionAll SetOperationNode.
  children: RelationalPlanNode[] = []
  collectUnionAllChildren(node, children)
  if children.length < tuning.parallel.minBranches: return null

  // Gate 1: every child concurrencySafe.
  if (children.some(c => c.physical?.concurrencySafe !== true)) return null

  // Gate 2: at least one child has expectedLatencyMs ≥ threshold.
  // (max-of-children matches the propagation rule the fanout rule established.)
  const maxLatency = Math.max(...children.map(c => c.physical?.expectedLatencyMs ?? 0))
  if (maxLatency < tuning.parallel.gatherThresholdMs) return null

  // Build the gather. preserveAttributeIds = original top SetOp's attributes
  // (which already mirror children[0].attributes per set-operation-node:32).
  concurrencyCap = Math.min(tuning.parallel.concurrency, children.length)
  return new AsyncGatherNode(
    node.scope,
    children,
    { kind: 'unionAll' },
    concurrencyCap,
    node.getAttributes(),
  )

collectUnionAllChildren(n, out):
  if n.nodeType === PlanNodeType.SetOperation && n.op === 'unionAll':
    collectUnionAllChildren(n.left, out)
    collectUnionAllChildren(n.right, out)
  else:
    out.push(n)
```

#### Why post-optimization

The rule must read `physical.expectedLatencyMs` and `physical.concurrencySafe`, which are populated as part of physical selection. Run in `PassId.PostOptimization` at a priority **after** physical-selection has finalized leaf properties but **before** materialization advisory (so the advisory doesn't double-wrap the gather's children). Suggested priority: 17 (between the `monotonic-range-access` cluster at ~9-10 and `materialization-advisory` at 20). Justify in the registration comment.

#### Why flatten the whole chain in one rewrite

Recursive single-pair fusion (collapse SetOp(A, SetOp(B, C)) → SetOp(A, B, C)... wait, SetOp is binary) would require multiple rule firings and produce intermediate states the optimizer must re-traverse. Doing it in one shot — building the full child list via `collectUnionAllChildren` — is one rewrite, one rule firing, simpler invariant.

#### Idempotence

A second firing of the rule on an `AsyncGatherNode` does nothing (the top node is no longer `SetOperation`). The rule's matcher already gates on `nodeType === PlanNodeType.SetOperation`. Mirror the idempotence test pattern from `rule-fanout-lookup-join.spec.ts`.

#### Attribute-ID preservation

`SetOperationNode.buildAttributes()` returns `left.getAttributes()` verbatim (`set-operation-node.ts:32-34`). For a left-deep chain of unionAll-SetOps, this propagates down to the leftmost child's attributes. Passing `node.getAttributes()` as `preserveAttributeIds` to the gather constructor keeps the consumer-visible attribute IDs identical pre/post rewrite. Lock with a test (snapshot Project's output attribute IDs around the rewrite).

### Interaction with existing rules

- **`SetOperationNode.computePhysical`** (`set-operation-node.ts:55-76`) already drops `monotonicOn`, `fds`, `equivClasses`, `constantBindings`, `domainConstraints` for unionAll. The gather node makes the same drops, so downstream rules see identical physical properties pre/post rewrite. No churn expected.
- **Predicate pushdown / FD-driven rules** run in earlier passes and operate on logical `SetOperation` nodes — they never see the gather. No interaction.
- **`Sort` directly above the chain.** A top-level `... order by x` over a unionAll chain wraps the SetOp tree in a `Sort`. The rule rewrites the SetOp child; the `Sort` continues to do its job above the gather. No change to result ordering.
- **`materialization-advisory`** runs at PostOptimization priority 20 — after the gather rule fires at 17. The advisory sees the gather's children individually (the gather node itself is opaque to the advisory's recognition); each child is evaluated for caching on its own merits. Verify with a test that an advisory-eligible child does still get wrapped in `CacheNode` *inside* the gather. If that's undesirable for the parallel pipeline (a cache wraps a remote source whose latency is exactly what we wanted to overlap), the advisory needs a skip-when-under-gather guard — file as a follow-up if the test surfaces the conflict.

### Golden-plan sweep

By the cost gate's design, no memory-vtab plan should change. The gather rule never fires in local-only configurations because `expectedLatencyMs` is zero throughout. Confirm with a clean run of the existing `test/plan/` and `test/optimizer/` goldens — **expect zero changed goldens**. If any change, the cost gate is leaking; fix the leak rather than blessing the diff.

## Tests

`packages/quereus/test/optimizer/parallel-async-gather.spec.ts` (new). Mirrors the fanout rule's test shape.

Coverage:

- **Synthetic high-latency vtab fixture.** Reuse the helper that the `4.5` fanout-rule spec builds (`getBestAccessPlan` returning `expectedLatencyMs: 25`). If `4.5` doesn't extract it to a shared test util, extract here.
- **`A UNION ALL B UNION ALL C` with all three high-latency → rewrite.** Assert the resulting plan contains exactly one `AsyncGatherNode(unionAll)` with `children.length === 3` and no remaining `SetOperationNode(unionAll)` nodes in the rewritten subtree.
- **`A UNION ALL B` (N=2) → rewrite when minBranches=2.** Pin the minBranches lower bound.
- **Single child after disqualification → no rewrite.** `A UNION ALL B UNION ALL C` where only A clears the threshold; B and C are local. The rule's gate is "max child ≥ threshold" — *all* children must be `concurrencySafe`, but `expectedLatencyMs` only needs the max to clear. Since `max(25, 0, 0) ≥ 25`, the rule still fires. Lock this — it's the "mixed branches" decision (same as the fanout rule's mixed-branches case).
- **One branch non-concurrencySafe → no rewrite.** Inject a child with `concurrencySafe: false` (e.g. an Insert sub-plan or a 'serial' module read with no per-branch connection). Assert the chain stays as `SetOperationNode`s.
- **All local → no rewrite, regardless of N.** Five-way memory-vtab unionAll, `expectedLatencyMs = 0` throughout. Rule must not fire. Pin by name to lock the "never on local" invariant.
- **Disabled via `tuning.disabledRules`.** With the rule disabled, the plan keeps the chained `SetOperationNode`s. Used as the baseline for the equivalence check below.
- **End-to-end execution equivalence.** Run a 3-way SQL `UNION ALL` over the synthetic high-latency vtab with rule enabled vs disabled; assert identical row multisets (no ordering claim).
- **Attribute-ID preservation.** Snapshot the top Project's output attribute IDs before optimization; assert they are unchanged after the rewrite.
- **Idempotence.** Apply the rule twice (manually drive the optimizer over the already-rewritten plan); assert no second-pass change.
- **Sort above the chain.** `select * from (A union all B union all C) order by x` — the gather appears under the Sort. Total result is correctly ordered (test asserts ordered equality).
- **Recursive CTE unaffected.** `with recursive r as (... union all ...) select * from r` — recursive CTEs use a distinct compound shape (`RecursiveCTENode`, not chained `SetOperationNode`s — see `recursive-cte-node.ts`). Assert the rule doesn't touch the recursive case.
- **`UNION` (distinct) ignored.** `A UNION B` (no ALL) → no rewrite. Pin the `op === 'unionAll'` matcher.
- **Golden-plan sweep.** Run the full existing `test/plan/` and `test/optimizer/` suites with the rule enabled. **Expectation: zero changed goldens.** If any change, the cost gate is broken — file the diff before blessing.
- **Strict-fork compatibility.** Run the spec once under `QUEREUS_FORK_STRICT=1`. The gather node's emitter is already strict-fork compatible (verified in ticket 5); this test confirms the rule-emitted plan also is.

Whether this is a *win* depends on the slow branch's actual latency vs. the overhead of forking three contexts when two will finish in microseconds. v1 takes the simple rule "max ≥ threshold"; adaptive per-branch decision is parked.
