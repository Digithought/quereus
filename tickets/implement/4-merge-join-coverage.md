`src/runtime/emit/merge-join.ts` currently has **17% statement coverage with 0 functions
called** — the emitter is effectively dead code in tests. The existing
`83-merge-join.sqllogic` file tests merge-join **semantics** by joining on non-PK columns
(`val`), which requires sorting both inputs. With default row estimates, hash join wins the
cost comparison, so the rule never picks merge join and the emitter is never invoked.

**Cost math (from `cost/index.ts`) for N rows each side:**
- Nested loop: `N + 0.1*N²`
- Hash join: `1.2*N`
- Merge join (sorted): `0.6*N`
- Merge join (needs sort): `0.6*N + 2*sortCost(N)`

Merge wins over hash **whenever both inputs are already ordered on the equi-join keys**.

**The fix: add tests that join on primary keys.**

When two memory tables are joined on their PKs and the access-path rule picks the PK index
scan for both sides, `isOrderedOnEquiPairs` returns true for both, `needsSort` is false, and
merge join wins cleanly: `0.6*N` vs hash's `1.2*N`.

Minimal example:
```sql
create table a (id integer primary key, x text);
create table b (id integer primary key, y text);
insert into a values (1,'a'),(2,'b'),(3,'c');
insert into b values (1,'x'),(2,'y'),(3,'z');
select * from a join b on a.id = b.id;
```

Plus multi-column PK joins and the LEFT/SEMI/ANTI variants to hit the join-type branches in
the emitter:
```sql
create table a (k1 int, k2 int, v text, primary key (k1, k2));
create table b (k1 int, k2 int, w text, primary key (k1, k2));
select * from a join b on a.k1 = b.k1 and a.k2 = b.k2;
select * from a left join b on a.k1 = b.k1 and a.k2 = b.k2;
```

**Verification**: add a plan-shape assertion (see companion `4-plan-shape-tests` ticket) that
these queries produce a `MergeJoin` in the plan. If they don't, the issue is that the access
path rule isn't promoting the PK scan to an ordered index scan — investigate
`rule-select-access-path.ts` ordering propagation.

**Fallback if PK path doesn't work**: use stats to force the decision. `ANALYZE` populates
`table.statistics.rowCount` via `catalog-stats.ts`. At large row counts merge join beats hash
even with dual sort. A test that seeds enough rows and runs ANALYZE first would reach the
sorted-merge path. This is messier and slower but a valid backup.

**Also investigate**: `sequencing.ts` (29% coverage, 0 funcs) and `retrieve.ts` (47%, 0 funcs)
have the same "never invoked" problem. `sequencing.ts` backs sequence-column node types that
may never be generated by current SQL; `retrieve.ts` is the retrieve-node fallback emitter
that's normally optimized away. Either add tests that exercise them or document them as
intentional fallback/dead code and exclude from coverage targets.
