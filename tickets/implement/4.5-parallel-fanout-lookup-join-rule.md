## Goal

A `Project`-rooted (and optionally `Aggregate`-rooted — see Open question below) recognition rule that:

1. Walks a chain of LEFT/INNER joins from a common outer.
2. For each join, verifies the non-preserved side is a parameterized FK→PK lookup matching the same shape `ruleJoinElimination` already accepts (AND-of-column-equalities ON clause, FK on the preserved side aligning to PK on the lookup side).
3. Clusters ≥ `tuning.parallel.minBranches` such joins into one `FanOutLookupJoinNode`.
4. Gates the rewrite on a cost comparison: projected latency savings (`(N − concurrencyCap) × expectedLatencyMs`) must exceed total branch setup overhead (`N × branchSetupCost`). For local-only plans (`expectedLatencyMs === 0`), the gate must never fire.

This rule depends on the node landing first (`4-parallel-fanout-lookup-join-node`). The single golden-plan sweep is performed *in this ticket's commit* — by then the node is committed and emittable, so the diffs are limited to the rule's output.

## Architecture

### New tuning knobs

In `optimizer-tuning.ts`, add a `parallel` section to `OptimizerTuning`:

```ts
readonly parallel: {
  /** Don't form a fan-out below this branch count. Default 2. */
  readonly minBranches: number;
  /** Per-branch fixed overhead, charged against the latency win. Default: pick from cost-constants — see TODO. */
  readonly branchSetupCost: number;
  /** Static cap on in-flight branches per outer row. Default 8. */
  readonly concurrency: number;
};
```

Add defaults to `DEFAULT_TUNING`. `minBranches: 2`, `concurrency: 8`. For `branchSetupCost`, pick a value comparable to `COST_CONSTANTS.NL_JOIN_PER_OUTER_ROW` × small-N — the value matters only relative to `expectedLatencyMs`, so anchor it to existing cost units (do not invent a new unit).

### New physical-properties fields

In `plan-node.ts`'s `PhysicalProperties`:

```ts
/**
 * Expected first-row latency in milliseconds for this subtree's iterator.
 * 0 for local-only paths (memory vtab, in-process compute). Non-zero for
 * remote vtabs and any operator whose cost model declares it. Consumed by
 * rule-fanout-lookup-join's cost gate; consumers must not rely on it for
 * correctness (only as a fan-out savings hint).
 */
expectedLatencyMs?: number;

/**
 * True when the subtree is safe to execute concurrently with siblings
 * sharing the same vtab connection. Defaults to true for read-only
 * subtrees over modules with concurrencyMode !== 'serial'. False where
 * the subtree mutates state, holds a non-reentrant cursor, or sits over
 * a 'serial' module that does not have a per-branch connection
 * available.
 */
concurrencySafe?: boolean;
```

Propagate both via the existing pass-through rules in `framework/physical-utils.ts`:

- `expectedLatencyMs`: take the **max** of children's values (the slowest child gates first-row latency). Memory-vtab leaves contribute 0. Remote-vtab leaves contribute a non-zero value sourced from the module's access plan (deferred to remote-vtab cost-model work — for now, conservatively read it from a yet-to-be-added `getBestAccessPlan` field or default to 0; document that until a real plugin populates it, the cost gate stays inert by design, which is the desired v1 behavior).
- `concurrencySafe`: AND of children — any non-safe child poisons the parent. Leaves derive theirs from `getModuleConcurrencyMode(vtabModule) !== 'serial' && readonly === true`.

### Recognition rule

`packages/quereus/src/planner/rules/join/rule-fanout-lookup-join.ts`. Mirrors `rule-join-elimination.ts`:

```
ruleFanOutLookupJoin(node, context):
  if not (node instanceof ProjectNode): return null
  walk the chain (Filter/Sort/LimitOffset/Distinct/Alias pass-through) collecting demanded attrs,
    bottoming out at the deepest Join. Then walk further down through *additional* Join nodes
    on the *left* (outer) side as long as each:
      - is LEFT or INNER joinType,
      - has an AND-of-column-equalities ON clause (reuse isAndOfColumnEqualities),
      - has the join's right side (the "lookup") aligned FK→PK against an outer column,
        verified via the existing `lookupCoveringFK` + `checkFkPkAlignment` machinery (so
        the at-most-one property the runtime relies on is *the same property*
        `ruleJoinElimination` already trusts).
  Stop the walk when:
      - the next operator down is not a Join,
      - or the next Join doesn't satisfy the above,
      - or the next Join's left side is a different relational subtree than the one we've
        accumulated (the "common outer" requirement breaks).
  Now we have:
      branches: [{ child, mode, outputAttrs, concurrencySafe }, ...]   // N entries
      outerSubtree: the relational node at the bottom of the chain (the deepest left side)
      chainOnTop: the pass-through wrappers above the cluster (Filter/Sort/Limit/Distinct/Alias)
      topProject: the Project node we started from
  if branches.length < tuning.parallel.minBranches: return null
  // Cost gate — must skip when local-only.
  totalLatency = max(branches.map(b => b.child.physical?.expectedLatencyMs ?? 0))
  if totalLatency === 0: return null
  concurrencyCap = min(tuning.parallel.concurrency, branches.length)
  savings = (branches.length - concurrencyCap) × totalLatency
  overhead = branches.length × tuning.parallel.branchSetupCost
  if savings <= overhead: return null
  // Build the FanOutLookupJoinNode preserving the Project's output attribute IDs.
  // For each branch, compute outputAttrs by walking the original join's right-side
  // attributes through any AliasNode the branch sits under.
  fanout = new FanOutLookupJoinNode(scope, outerSubtree, branches, concurrencyCap, preserveAttrs)
  // Rebuild the chain wrappers above the fanout, then wrap in the Project, mirroring
  // ruleJoinElimination's rebuildChain + rebuildProject helpers.
  return rebuildChain(chainOnTop, fanout) wrapped in rebuildProject(topProject, ...)
```

The branch's `mode` is the original join's type: `LEFT → 'atMostOne-left'`, `INNER → 'atMostOne-inner'`. The INNER case additionally requires the `lookupCoveringFK` nullability check (matches `ruleJoinElimination`'s tryEliminate's INNER guard).

Per-branch `concurrencySafe` is derived from the branch's child subtree's propagated physical property.

The non-equi-residual restriction is identical to `ruleJoinElimination` (`isAndOfColumnEqualities`). Any residual disqualifies the branch — that branch stays as a normal nested-loop join, and the rule either clusters fewer branches (≥ minBranches) or returns null.

### Golden-plan sweep

The cost gate's `expectedLatencyMs === 0` skip means: with no remote-vtab plugin in tree, the rule does not fire. Memory-vtab tests therefore see **no** plan-shape changes. This deliberately keeps the v1 golden-plan churn near zero.

Validate this with the test sweep below. If, contra expectation, any memory-vtab golden actually does change, the cost gate is wrong (or `expectedLatencyMs` is leaking non-zero from somewhere). Fix the leak rather than blessing the diff.

For *positive* coverage where the rule actually fires, the recognition tests in `test/optimizer/parallel-fanout.spec.ts` build plans against a synthetic vtab whose access plan reports `expectedLatencyMs > 0`. See the test list below.

## Tests

`packages/quereus/test/optimizer/parallel-fanout.spec.ts` (new). Covers recognition and cost-gate behavior; the runtime correctness tests already live in the node ticket.

Coverage shape:

- **Synthetic high-latency vtab module.** Helper that produces a TableSchema + vtab module whose `getBestAccessPlan()` returns `expectedLatencyMs: 25`. Used as the lookup table in multi-branch scenarios.
- **Below `minBranches` → no rewrite.** A single LEFT join (N=1) over the high-latency module: assert the plan does not contain `FanOutLookupJoin`.
- **At `minBranches` and above → rewrite fires when latency wins.** A 3-branch chain (orders → 3 lookups): assert `FanOutLookupJoin` appears with `branches.length === 3` and the original joins are gone.
- **High-latency, but `concurrency` cap leaves no savings → no rewrite.** Synthetic case where `concurrency >= N` so `(N − cap) × latency = 0` → cost gate rejects. Pin: confirm rule returns null and the chain is unchanged.
- **Local-only chain → no rewrite, regardless of N.** A 5-branch chain over the memory vtab (latency=0): rule must not fire. Pin by name to lock the "never on local" invariant.
- **Mixed branches (some local, some remote).** Latency is the *max* of branches. Confirm the rule still clusters when the max is non-zero; document whether local branches share the fan-out (recommended: yes — they pay branchSetupCost but their `expectedLatencyMs` of 0 doesn't reduce the max savings).
- **Residual non-equi predicate on one branch.** Branch with `ON l.fk = r.pk AND r.x > 5` → that branch stays out of the cluster; if the remaining branches are still ≥ `minBranches`, the fan-out forms over them.
- **INNER branch with nullable FK.** Disqualified via the same nullability check `ruleJoinElimination` uses; branch stays as a nested-loop join.
- **Attribute-ID preservation.** Rule rewrites the chain, but the top Project's output attribute IDs are unchanged (parent rules / consumers must still see the same IDs). Walk the rewritten tree, find the new Project, and assert its `attributes` IDs match the pre-rewrite snapshot.
- **End-to-end execution equivalence.** Run a 3-branch SQL query against the synthetic high-latency vtab with the rule enabled vs. disabled (`tuning.disabledRules`), and assert identical row sets.
- **Golden-plan sweep.** Run the full existing `test/plan/` and `test/optimizer/` suites. **Expectation: zero changed goldens.** If goldens change, the cost gate is broken — file the diff before deciding to bless it.

- Connection-per-branch acquisition is not implemented; v1 always shares the outer's connection. `'serial'`-mode branches serialize via the lock and get no concurrency win — correctness is preserved but the parallelism payoff is module-mode-gated.
- Adaptive concurrency, latency-driven branch ordering, and full FD-propagation primitives for the multi-branch shape are explicitly out of scope per the plan ticket.
