This is the **second half** of the public change-scope API. Depends on
`fd-public-change-scope-data-and-analyzer` having landed the `ChangeScope`
data contract, the `analyzeChangeScope` function, and the public re-exports.
Without that ticket, the data shape consumed by the watcher does not exist.

The watcher is **plan-independent**: it consumes a `ChangeScope` value
regardless of provenance — freshly analyzed, deserialized from disk,
hand-built in test code, or received over a network. This decoupling is the
whole point of the two-halves design and must be visible in the test
matrix.

## Architecture

### Public API

In `packages/quereus/src/core/database.ts`:

```typescript
public watch(scope: ChangeScope, handler: WatchHandler): Subscription;
```

New types live in `change-scope.ts` next to the data contract (so the
public API surface is one module):

```typescript
export interface Subscription {
  readonly id: string;
  unsubscribe(): void;
}

export interface WatchEvent {
  readonly matched: ReadonlyArray<MatchedWatch>;
  readonly txnId: string;
}

export interface MatchedWatch {
  readonly watch: TableWatch;
  /** For 'rows' / 'rowsByGroup' watches: keys/groups in this txn that hit.
   *  For 'groups': distinct group keys touched.
   *  For 'full': empty array (table touched, no narrowing). */
  readonly hits: ReadonlyArray<ReadonlyArray<SqlValue>>;
}

export type WatchHandler = (event: WatchEvent) => void | Promise<void>;
```

### Implementation

`Database.watch` translates the public `ChangeScope` into the existing
internal `DeltaSubscription` shape and registers it against the database's
`DeltaExecutor` (the same kernel assertions and MV maintenance use). The
translation is mechanical and lives **in `delta-executor.ts`** as a new
exported helper `subscriptionFromChangeScope(scope, handler, ctx)` so the
shape mapping is testable in isolation:

| `WatchScope.kind` | Internal `BindingMode` | Notes |
| --- | --- | --- |
| `full`            | `{kind: 'global'}`                       | Optional column narrowing — see below. |
| `rows`            | `{kind: 'row',   keyColumns: <idx>}`     | Pre-filter `getChangedTuples` output by `values`. |
| `groups`          | `{kind: 'group', groupColumns: <idx>}`   | Hits = distinct group keys touched. |
| `rowsByGroup`     | `{kind: 'group', groupColumns: <idx>}` + row pre-filter | Treat as group binding for kernel; intersect with `values` post-hoc when forming hits. |

Mapping column names → column indices uses the existing schema lookup on
the base table (`schemaManager.findTable`). If the table or any column in
the scope no longer exists at watch-registration time, `watch` throws a
`QuereusError` (clear message: which table, which column). After
registration, schema changes invalidate the subscription; on
`SchemaChangeEvent`, dispose the subscription and call the handler with
a synthetic event signaling invalidation? **No — for v1**, on schema
change just dispose and log a warning. Re-subscribing is the caller's
responsibility. Document this in `docs/change-scope.md`.

The `apply(input)` callback inside the synthesized `DeltaSubscription`:

1. Builds the `MatchedWatch[]` array by iterating `scope.watches` and
   matching each against `input.perRelationTuples` / `input.globalRelations`.
2. For `rows`/`rowsByGroup`, intersect the kernel's tuples with
   `scope.watches[i].scope.values` (literal-only; `ScopeValue.param`
   placeholders are an error here and indicate the caller forgot to call
   `bindParameters`. Throw a clear error at registration time if any
   `unboundParameters` remain — see "Validation" below).
3. Calls `handler({matched, txnId})`. The `txnId` comes from the
   `DeltaExecutorContext` — extend `DeltaExecutorContext` with a
   `getCurrentTxnId(): string` method (or inject through the runOne
   call site if cheaper). If no stable txn id is available today,
   generate a monotonic counter inside the `Database`'s registration
   site and document the limitation.
4. Handler errors are caught and logged; they do NOT roll back the
   commit (assertions own that contract, watchers do not). Document
   this divergence prominently.

### Column tracking on `full` watches

A `{kind: 'full', columns: {a, b}}` scope means "watch the whole table
but only columns `a` and `b` matter." For the kernel, this is still a
`global` binding (no per-row narrowing), but the watcher post-filters:
when forming the `MatchedWatch` for a `full` watch with a non-`'all'`
column set, only fire if the captured changes touched at least one of
those columns. Implementing this requires the watcher to register a
`registerCaptureSpec` for the named columns at watch time so the
TransactionManager actually captures column-level deltas (mirrors the
assertion path). If `columns: 'all'`, no capture spec is registered;
fire on any change.

### Validation

`watch(scope, handler)` rejects (throws `QuereusError` synchronously) if:

- `scope.unboundParameters.length > 0` — the kernel cannot bind values
  from `ScopeValue.param` placeholders. Caller must run
  `bindParameters(scope, params)` first. Error message must say so.
- Any `TableWatch` references a table that does not exist in the schema.
- Any column referenced in `key`, `groupBy`, or `columns` (when not
  `'all'`) does not exist on its table.

`watch(scope, handler)` accepts (and warns once via `log.warn`) if:

- `scope.watches.length === 0` and `scope.nonDeterministicSources.length === 0`
  — dead subscription, will never fire. Log includes the `Subscription.id`
  so the caller can find it. Useful for tests; not an error.

`scope.nonDeterministicSources` is **advisory metadata only**. The
watcher does not synthesize fake events for time/random sources. If the
caller needs polling for time-sensitive queries, that is outside this
ticket. Document this clearly.

### Firing semantics

- After-commit only. Mirrors assertion COMMIT eval and MV maintenance.
- Mid-transaction changes do not fire.
- Multi-table scope fires once per transaction with all relevant
  `MatchedWatch` entries in the same `WatchEvent`.
- Empty `matched` array → do not fire (no observable change).
- Handler is `void | Promise<void>`. If a handler returns a Promise,
  the watcher awaits it before processing the next subscription, but
  rejection is logged and swallowed (does not roll back the commit).

### `Subscription.id`

Stable hash of the canonical (sorted) scope + a random nonce, formatted
as `watch:<base32-hash>:<nonce>`. Useful in logs and for cross-process
correlation when a scope is shared.

## Open design points (defaults are committed)

- **Schema change → dispose**, not auto-rebind. Documented; v1 keeps it
  simple.
- **Handler errors do not roll back the commit.** Asymmetric with
  assertions; the contract is "watchers observe, assertions enforce."
  Documented.
- **`txnId` shape.** Default: opaque string from a monotonic counter
  inside `Database` if no transaction-manager-assigned id exists.
  Stable within one `Database` instance; not portable across processes.
- **Re-entrancy.** A handler that calls `database.watch` or
  `subscription.unsubscribe` during a fire is legal; the kernel iterates
  a snapshot of subscriptions per commit, so the new/removed
  subscription takes effect on the next commit, not the current one.
  Add a one-line note in `docs/change-scope.md`.

## Tests

### Watcher integration (`test/logic/change-scope.spec.ts`, extending
the file from the prior ticket)

- Hand-built `ChangeScope` (no `Statement`, no analyzer) registers and
  fires correctly. **This is the proof that the watcher is
  plan-independent — gate the ticket on this test.**
- `rows(pk, [7])` watch fires on update of row 7 with `hits: [[7]]`,
  does not fire on update of row 8.
- `groups([customer_id])` watch fires once per distinct group key
  touched in a transaction (multiple rows in the same group → one entry
  in `hits`).
- `full` watch with `columns: 'all'` fires on any row mutation,
  `hits: []`.
- `full` watch with `columns: {a}` fires only when column `a` is
  mutated (touch column `b` only → no fire).
- `unsubscribe` stops further firings; `unsubscribe` is idempotent.
- Multi-table scope fires once per matching transaction with all
  relevant `MatchedWatch` entries in the same event.
- End-to-end: `db.prepare(sql).getChangeScope([7])` →
  `db.watch(scope, handler)` → mutate matching row → handler fires with
  the expected hits.
- Validation: `db.watch(scopeWithUnboundParams, h)` throws synchronously
  with a message that mentions `bindParameters`.
- Validation: `db.watch(scopeReferencingMissingTable, h)` throws with
  the table name.
- Dead subscription: `db.watch({watches:[], nonDeterministicSources:[],
  unboundParameters:[]}, h)` succeeds, logs a warning, never fires.

### Watcher unit (`test/runtime/delta-executor-watch.spec.ts`)

Hits the `subscriptionFromChangeScope` helper directly without touching
the runtime, asserting the translation table above column-by-column for
each `WatchScope.kind`. Test that a `rowsByGroup` scope produces a
`group` `BindingMode` with the right `groupColumns`, and that `apply`
intersects against `values` correctly when synthesizing `hits`.

### Property test (extend optimizer property suite)

For random simple queries: analyze → bind → watch → mutate a column not
mentioned in the scope → assert handler did not fire. Mutate a column
in the scope → handler may fire (not asserting it does — coarse but
catches false-negative regressions). Keep the suite size small (≤25
queries) so it stays inside the agent timeout.
