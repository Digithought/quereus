### Why this is pre-existing

This ticket only touched:

- `docs/runtime.md` — added one sentence on the no-lock-target fall-through.
- `packages/quereus/test/runtime/fanout-lookup-join.spec.ts` — added two tests for empty-outer-source and branch-error propagation.

Neither of these can influence the `distinct-elimination` optimizer rule or its differential test. The failing test is a property-based fuzz (`fast-check`) over random SQL queries asserting that `select distinct …` returns identical results with and without the `distinct-elimination` rule enabled. The counterexample is a `LIMIT 48` query whose DISTINCT result set is sensitive to row order vs LIMIT placement — that's an interaction inside the planner's distinct + filter + limit pipeline, not the parallel runtime.

The full suite ran clean (3384 passing, 9 pending, 0 failing) twice during this review on the same HEAD before the test additions, and strict-fork mode ran clean (3393 passing) — different fast-check seeds, so the failure is stochastic in this test, not a regression introduced here.
