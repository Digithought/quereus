# Pre-existing (flaky) test failure

## Command
From repo root:
```
yarn workspace @quereus/quereus run test
```
(package `@quereus/quereus`, Mocha suite — the differential fuzz spec
`packages/quereus/test/fuzz.spec.ts`.)

## Failing test
`Optimizer Equivalence › predicate pushdown rules produce identical results`

Excerpt:
```
Error: Property failed after 23 tests
Counterexample: [{"tables":[{"name":"t1","columns":[
  {"name":"c_blob0","type":"blob","notNull":true,"unique":true},
  {"name":"c_blob1","type":"blob","notNull":true,"unique":true,"primaryKey":true}]}]},6]
Caused by: Error: Row count mismatch: 1 (full) vs 0 (restricted)
  when disabling rules [predicate-pushdown, filter-merge]
  SQL: select distinct cast('test string' as text) as col0 from t1
       where 1 = 1 group by c_blob1 having c_blob1
```

## Why it is pre-existing (not from this ticket)
- This ticket only adds a guard to `ruleOrderByFdPruning`, which rewrites
  `SortNode`s exclusively. The counterexample query has **no ORDER BY**, and the
  seeded table has 6 rows, so there is no `SortNode` over a provably ≤1-row
  source anywhere in its plan — the new code path cannot run for this query.
- The mismatch is between the full rule set and a set with
  `[predicate-pushdown, filter-merge]` disabled — neither is the rule this
  ticket touches. The root cause looks like blob truthiness in a `HAVING`
  clause (`having c_blob1`) being evaluated differently depending on whether
  the predicate is pushed down/merged.
- The test uses fast-check with **random seeds** (no fixed seed; `numRuns: 25`).
  It is flaky: the predicate-pushdown differential test passed **20/20** runs on
  `main` (stash applied) and **20/20** runs with this ticket's changes applied.
  The single failure surfaced once during a full-suite run.

## Work-around
None applied — the failure is intermittent and does not block this ticket's
own tests (build + `rule-orderby-fd-pruning.spec.ts` + `monotonic-limit-pushdown.spec.ts`
all pass). No rules were disabled.
