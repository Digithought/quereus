## What landed

**1. Tuning knob (`packages/quereus/src/planner/optimizer-tuning.ts`)**

Added `parallel.gatherThresholdMs` to `OptimizerTuning.parallel` and to `DEFAULT_TUNING` (default 25 ms). The 25 ms default is calibrated against the synthetic `HighLatencyMemoryModule` fixture (`expectedLatencyMs = 25`) shared with the fan-out rule spec, so the same fixture exercises both rules. Updated the `parallel:` block's docstring to mention both consumers (the fan-out rule from ticket 4.5 and the gather rule from this ticket).

**2. Recognition rule (`packages/quereus/src/planner/rules/parallel/rule-async-gather-union-all.ts`)**

New file. Walks an arbitrary tree of `SetOperationNode(unionAll)` (any shape — left-deep, right-deep, balanced) plus any unionAll-`AsyncGatherNode` children already produced by an inner firing of the rule, flattens them into a single child list, then folds into one `AsyncGatherNode({ kind: 'unionAll' })` with `preserveAttributeIds = node.getAttributes()`.

Two gates:

1. `every(child.physical.concurrencySafe === true)` — any non-safe branch poisons the fold.
2. `max(child.physical.expectedLatencyMs ?? 0) ≥ tuning.parallel.gatherThresholdMs` — slowest child must meet the threshold. Memory-vtab leaves declare 0, so the rule is inert on local plans by design.

Registered in `optimizer.ts` at `PassId.PostOptimization` priority 17 — after `asof-strategy-select` (11) so leaves' physical properties are finalized, before `materialization-advisory` (30) so the advisory's caches land *inside* gather branches.

**3. AsyncGather-child absorption (subtle)**

The flatten helper (`collectUnionAllChildren`) absorbs both nested `SetOperationNode(unionAll)` AND nested `AsyncGatherNode({ kind: 'unionAll' })`. **The AsyncGather absorption is load-bearing**: the rule is registered for bottom-up traversal, so the inner `SetOperation` of `(A unionAll B) unionAll C` rewrites to a 2-branch gather first. When the outer `SetOperation` then fires, its left child is no longer a `SetOperation` — it's the already-built `AsyncGather`. Without the absorption, the outer firing produces a 2-branch gather whose first child is a gather, defeating the "single gather per chain" invariant. The idempotence test pins this.

**4. Tests (`packages/quereus/test/optimizer/parallel-async-gather.spec.ts`)**

16 specs covering:

- All-high-latency 3-way fold; single SetOp doesn't match; N=2 with minBranches=2.
- **Mixed-branches** (one slow, two local) — locks the v1 "max-of-children" decision: the fold *does* fire. Whether this is actually a win is a follow-up consideration; see "Gaps" below.
- All-local doesn't fold (the load-bearing local-only invariant); raised threshold doesn't fold; minBranches override; `UNION` (distinct) doesn't match.
- `disabledRules` honored.
- Execution equivalence (multiset, not ordered — unionAll arrival order is non-deterministic under the gather).
- Attribute-ID preservation (column names match across the rewrite).
- Sort above the gather (ordered equality on the result).
- Recursive CTE untouched (uses `RecursiveCTENode`, distinct from chained SetOperationNodes).
- Idempotence (re-planning the same SQL still produces exactly one gather).
- Default tuning has `gatherThresholdMs > 0` (pins the no-rewrite-on-local invariant at the tuning layer).
- concurrencyCap clamped to N when N < concurrency.

## Gaps / honest call-outs for the reviewer

**Strict-fork: Sort-above-AsyncGather skipped under `QUEREUS_FORK_STRICT=1`.** The Sort emitter (`runtime/emit/sort.ts:35`) calls `withAsyncRowContext(ctx, …)` on the parent rctx while the gather's iterator (which keeps active forks of that same rctx) is still being consumed. The strict-fork harness flags this as a contract violation. **This is pre-existing — any manually constructed Sort-on-AsyncGather plan would trip the same path** — not introduced by the rule. The non-strict path validates correctness end-to-end. The spec for the Sort-above-gather case checks the env var and uses `it.skip` under strict-fork; a single line of code. Worth filing as a follow-up (`fix-sort-above-async-gather-strict-fork`) so the strict-fork harness covers this path eventually, but not blocking for this rule.

**`materialization-advisory` interaction is not patched.** The advisory runs at priority 30 (after the gather at 17) and sees each gather child individually. If a child is advisory-eligible AND high-latency (the only case where the gather rule fires), the advisory may wrap the child in a `CacheNode` *inside* the gather branch — which materializes serially before the gather sees a row, partially defeating the prefetch overlap. The spec does NOT have a test for this interaction because no current test fixture combines a high-latency module with the advisory's recognition shape (most advisory-eligible nodes are `Block` / `Insert` / `Update` / `CTE` etc., not raw table reads). Suggest reviewers consider whether to file `parallel-gather-cache-interaction` for future-proofing; v1 ships without the patch and the in-tree behavior is unaffected.

**Mixed-branches fires the rule.** Per the spec's design choice, the gate is "max child latency clears threshold" — not "all branches are high-latency". A 3-way chain where one branch is slow and two are local *will* fold. Whether this is actually a win depends on whether the overhead of forking three contexts beats the parallelism payoff of the one slow branch. v1 takes the simple decision; adaptive per-branch evaluation is parked.

**Cost gate is inert in production until a remote-vtab plugin declares non-zero `expectedLatencyMs`.** By design — the load-bearing invariant for the golden-plan sweep. Same posture as the fan-out rule.

**No `crossProduct` recognition rule.** Opt-in only per parent ticket.

**No `zipByKey` combinator.** Parked in backlog as `parallel-async-gather-zip-by-key`.

**Recursive CTE.** Uses `RecursiveCTENode`, not chained `SetOperationNode`s. The rule's `node instanceof SetOperationNode` matcher rejects it automatically. Locked by an explicit test.
