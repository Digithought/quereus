# Review: incremental-MV change-scope projection

Adversarial review of the landed Phase-D projection. Treat the implementation as a
**starting point** and the tests as a **floor**. The behavioral contract is small
but load-bearing for reactive consumers: a `Database.watch` on an incremental MV
must fire on source mutations, and a manual MV must still report its backing table.

## What shipped (verify each against the code)

1. **`MaterializedViewSchema.sourceScope?: ChangeScope`** (`schema/view.ts`) — cached
   source-union change-scope for an `on-commit-incremental` MV. Type-only import of
   `ChangeScope` from `planner/analysis/change-scope.js` (erased at compile time — no
   runtime cycle). Documented as not-serialized / repopulated on re-registration.

2. **`buildSourceUnionScope(sourceTables)`** (`change-scope.ts`, composition-helpers
   section) — v1 conservative union: one `{kind:'full'}`, `columns:'all'` watch per
   source. Parses qualified `schema.table` via `indexOf('.')` with a `'main'` fallback
   (mirrors the `revalidateBody` parser in `materialized-view-helpers.ts`). Normalized
   via `normalizeWatches`.

3. **Cache population** (`database-materialized-views.ts`
   `MaterializedViewManager.registerMaterializedView`) — sets
   `mv.sourceScope = buildSourceUnionScope(mv.sourceTables)` right after the
   `on-commit-incremental` policy guard (no-op for manual). This is the single point an
   MV becomes incremental; the create emitter and the declarative `apply schema` path
   both reach it via `db.registerMaterializedView`.

4. **`SchemaManager.getMaterializedViewByBackingTable(schemaName, backingName)`**
   (`schema/manager.ts`) — O(1) reverse lookup: rejects non-`sqlite_mv_` names via the
   `backingTableNameFor('')` prefix, derives the MV name from the suffix, then confirms
   `mv.backingTableName.toLowerCase() === backingName.toLowerCase()`. Added a value
   import of `backingTableNameFor`.

5. **`MaterializedViewSourceResolver` + analyzer projection** (`change-scope.ts`) —
   `analyzeChangeScope` gained an optional `resolveMaterializedViewSource` option.
   In the per-ref loop, a resolved MV scope is collected into `mvSourceScopes` and the
   backing-table watch is skipped; after the base scope is built, each MV scope is
   folded in with `unionScopes` (so reading both the MV and a source directly reports
   the source once). Param binding still runs last. Gated on `!dmlWithoutReturning`.

6. **`Statement.getChangeScope` wiring** (`core/statement.ts`) — passes a resolver
   backed by `this.db.schemaManager.getMaterializedViewByBackingTable`, returning
   `mv.sourceScope` **only** when `refreshPolicy?.kind === 'on-commit-incremental'`.

7. **Docs** — `materialized-views.md` gained a "Change-scope projection" subsection and
   flipped the Limitations bullet from "tracked follow-up" to "delivered (conservative)";
   `change-scope.md` gained a "Materialized-view reference projection" subsection + a
   Known-imprecisions bullet; `optimizer.md` § Binding-aware Delta notes the projection.

## Use cases to exercise (validation)

- `select * from <incremental-mv>` → `getChangeScope().watches` reports the **source**
  table(s), NOT `sqlite_mv_*`. (spec: "an INCREMENTAL materialized-view reference
  reports the SOURCE table".)
- `select * from <manual-mv>` → still reports `sqlite_mv_*`. (spec: "a MANUAL
  materialized-view reference reports the BACKING table".)
- `db.watch(stmt.getChangeScope())` on an incremental MV → fires on a source INSERT/
  UPDATE/DELETE. (spec: "end-to-end: a watch on an INCREMENTAL MV fires on a SOURCE
  mutation".)
- Query joining the MV and its source → source appears exactly once (union/dedup).
- Multi-source: v1 only registers single-source bodies incrementally (the manager's
  eligibility gate rejects joins), so `sourceTables` is effectively length-1 in
  practice — but `buildSourceUnionScope` handles N sources for when that gate widens.

All new tests live in `test/logic/change-scope.spec.ts` (3 new + 1 sharpened). Suite:
`yarn test` in `packages/quereus` → **3756 passing, 9 pending** (pending pre-existing).
Lint + typecheck clean.

## Things to scrutinize / known gaps (flag, don't fix inline unless trivial)

- **Source-union is whole-table (`full`), not row/group-precise.** The ticket's
  v1 explicitly accepts this; the precise per-source scope (mirroring the maintenance
  bindings the manager already derives in `compile()`) is a documented refinement. A
  reviewer wanting precision should weigh reusing the manager's `perRelation`
  bindings vs. recomputing — but that is follow-up scope, not this ticket.
- **`sourceScope` is only populated at registration.** Quereus is in-memory and every
  MV is created through the emitter (which registers), and `apply schema` re-creates
  via the same path — so there is no live "deserialized catalog without re-registration"
  gap today. If a future persistence/load path adds MVs to the catalog *without*
  calling `registerMaterializedView`, an incremental MV would have `sourceScope ===
  undefined` and the resolver gracefully falls back to reporting the backing table
  (sound, just not projected). Confirm no such path exists / will exist.
- **Stale incremental MV.** On a source schema change the manager calls `releaseEntry`
  but does NOT clear `mv.sourceScope`; the MV is marked `stale`. `getChangeScope` on a
  stale MV either throws at plan-build (incompatible source change — same as executing
  the query) or, if revalidation passes, still projects to the cached (now-current)
  sources. No correctness issue found, but worth a second look at the
  source-dropped-then-watch interaction (the projected `full` watch would name a
  dropped table → `db.watch` would reject it, which is the same failure executing the
  MV gives).
- **Resolver runs per table-ref per `getChangeScope` call.** Cheap (prefix check + map
  get), but confirm there's no hot path calling `getChangeScope` in a tight loop where
  this matters.
- **Manual-MV semantics decision.** A manual MV's backing table *is* user-observable
  (refresh writes it) and its cadence is `refresh`, so it intentionally keeps reporting
  the backing table. Confirm this is the desired contract for a watcher that wants to
  know "when did the manual MV's content change" — today they'd watch `sqlite_mv_*`
  and see the refresh write. (This matches the ticket; calling it out as a contract
  choice, not a bug.)

## Out of scope (do not expand here)

Precise per-source row/group projection; multi-source/join incremental eligibility;
auto-rebind of watchers on schema change (a pre-existing change-scope v1 limitation).
