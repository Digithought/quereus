## What landed

Three tests in `packages/quereus/test/runtime/fork-contract.spec.ts` and a small env-gated production hook to back the third:

**1. Pinned-keys drift.** `EXPECTED_FORK_POLICY` declares a fork policy (`forked` / `shared-frozen` / `shared-sink` / `shared-cooperative`) for every field of `RuntimeContext`. The `satisfies Record<keyof RuntimeContext, ForkPolicy>` clause turns "added a field without thinking about fork semantics" into a TypeScript compile error in the test. Two sanity tests follow it (forked fields actually differ across siblings, shared fields actually alias parent).

**2. Mutation-site allowlist.** Two tests walk `packages/quereus/src/` for direct `tableContexts.set/delete` and `<rctx-receiver>.context.set/delete` and compare the matching files against hardcoded allowlists. Today's allowlists:

- `TABLE_CONTEXTS_MUTATION_ALLOWLIST`: `src/runtime/emit/recursive-cte.ts` only.
- `ROW_CONTEXT_MUTATION_ALLOWLIST`: `context-helpers.ts`, `emit/aggregate.ts`, `emit/hash-aggregate.ts`, `emit/window.ts`. The original ticket assumed only `context-helpers.ts` would appear here; the three aggregate/window emitters were also direct mutators in current code and were added to the allowlist deliberately. The regex restricts the row-context check to `src/runtime/` only, to avoid false-positives on `OptimizationContext.context` (a totally unrelated `Map<string, unknown>` in `planner/framework/context.ts`).

Stale-allowlist guard included: any entry that no longer matches the codebase fails the test, so the allowlist can't rot silently.

**3. Strict-fork test mode.** `QUEREUS_FORK_STRICT=1` flips on a Node-only proxy/subclass on every `RuntimeContext.tableContexts` and `RuntimeContext.context` constructed at the five production sites (`core/statement.ts`, `core/database.ts`, `core/database-assertions.ts`, `runtime/deferred-constraint-queue.ts`, `planner/analysis/const-evaluator.ts` Ă—2) plus the fork's own maps inside `ParallelDriver.fork()`. The wrapper throws `strict-fork: parent context mutated ... while N fork(s) are active` if any `set` / `delete` / `clear` runs on a parent map while one of its forks is being driven by `ParallelDriver.drive()`.

Design choices worth flagging:
- **Per-parent-map state, not global.** A `WeakMap<wrappedMap, { activeForks }>` records each wrapped Map's own counter. `fork()` stamps each child's map with a back-reference to its parent's state (`markForkOf`). `drive()` reads the back-reference off `forks[0]` and bumps/drops the parent's counter on entry/finally. This handles concurrent unrelated drivers, sub-forks-of-forks, and lets a fork freely mutate its own (fresh) maps without false-positives.
- **Bump in `drive()`, not `fork()`.** Matches the ticket â€” `fork()` can be called speculatively without a subsequent `drive()`. The harness only treats forks as "live" during their drive.
- **Cross-platform guard.** The `process.env` read is wrapped in `typeof process !== 'undefined'` so the file remains browser/RN/edge-worker safe; outside Node, strict mode is silently disabled. The `env-compat.spec.ts` cross-platform audit gated this.

**Wiring & tooling:**

## Strict-fork findings

- **`yarn test:fork-strict` passed on the unmodified codebase**: 3317 passing, 0 failing. The 4-test delta over the non-strict run (3313) is the four strict-only assertions inside `fork-contract.spec.ts`. No real fork-contract violations surfaced.
- **No `shared-sink` field needed promotion to `forked`.** The tracer / contextTracker / planStack policies in the table were sufficient â€” the existing parallel-driver.spec.ts already exercises forked contexts, and nothing in the broader suite reads from these in a way that would interleave under parallel branch execution. (There are no parallel consumers yet, so this is a clean-slate floor, not exhaustive proof.)
- **CI integration recommendation:** add `test:fork-strict` as a separate CI job alongside the existing `test` and (optional) `test:store` jobs. Gating *every* `test` invocation behind strict mode would double the test budget for marginal additional coverage â€” the value of the harness is catching regressions when a new consumer is added, and a dedicated job is the right granularity for that. If CI is out of scope here, the manual invocation should be run before #2 (`parallel-eager-prefetch-node`) starts.

## Riskiness

Low. The production-side hook is a single `typeof process !== 'undefined'`-guarded `process.env` read plus two factory-function call sites at each of five construction points. Non-strict mode collapses every helper to a one-line pass-through. The full test suite is unchanged in behavior (3313 â†’ 3313 passing in non-strict mode, modulo the 4 strict-only additions to the same file).

The one design point worth a second look is the choice to wrap forks' own maps (not just the parent's). The rationale is that a fork might itself be sub-forked, and we want the same protection. The cost is one Proxy/subclass allocation per fork â€” measurable only at extreme fan-out, which has no consumers today.
