## What landed

### New physical-properties fields (`nodes/plan-node.ts`)

Two optional fields on `PhysicalProperties`:

- `expectedLatencyMs?: number` — expected first-row latency. Default 0 (local-only paths). Non-zero only when a leaf populates it.
- `concurrencySafe?: boolean` — true when the subtree is safe under sibling concurrency on a shared connection.

Propagation in `PlanNode.physical` (the base getter that merges child physicals):
- `expectedLatencyMs = max(children's values)` (slowest child gates first-row latency)
- `concurrencySafe = AND(children)` (any non-safe child poisons the parent)

Leaf source: `TableReferenceNode.computePhysical` now sets:
- `concurrencySafe = getModuleConcurrencyMode(vtabModule) !== 'serial'`
- `expectedLatencyMs = vtabModule.expectedLatencyMs` when > 0

A new optional field `expectedLatencyMs?: number` was added to `VirtualTableModule` so plugins can declare a per-module first-row latency hint. The `MemoryTableModule` omits it (treated as 0).

### Tuning knobs (`optimizer-tuning.ts`)

```ts
tuning.parallel: {
  minBranches: number;       // default 2
  branchSetupCost: number;   // default 1.0 — anchored against NL_JOIN_PER_OUTER_ROW
  concurrency: number;       // default 8
}
```

### Recognition rule (`rules/join/rule-fanout-lookup-join.ts`)

Registered in `optimizer.ts` under `PassId.Structural` at **priority 23** — one ahead of `join-elimination` (24). Operates on `ProjectNode` and:

1. Walks down through pass-through wrappers (Filter / Sort / LimitOffset / Distinct / Alias) to the first `JoinNode`.
2. Walks down the `.left` spine collecting joins until a non-Join is reached. The deepest left subtree is the *outer*; it must resolve to a single base table via `extractTableSchema` (mirroring `ruleJoinElimination`'s requirement). Each join's `.right` is a candidate branch.
3. For each candidate branch, verifies:
   - `joinType` is LEFT or INNER, condition present.
   - `isAndOfColumnEqualities(condition)` (reused from `rule-join-elimination`).
   - FK→PK alignment via `checkFkPkAlignment` over outer's schema + the lookup's schema, after translating left-side equi-indices through the join's combined-attribute layout into outer-attribute IDs (this is the safety net for chains where the join's `.left` spans multiple joins).
   - INNER branches require `lookupCoveringFK` to find a non-null FK and `isRowPreservingPathToTable` on the lookup.
4. Bails out if branches < `tuning.parallel.minBranches`.
5. Cost gate: `(N − concurrencyCap) × max(expectedLatencyMs) > N × branchSetupCost`, where `concurrencyCap = min(tuning.parallel.concurrency, N)`. Clamps to 0 savings when `cap ≥ N` (fan-out wins only when concurrency-bound).
6. Builds `FanOutLookupJoinNode` with `preserveAttributeIds` pinning the outer-then-branches wide layout to exactly what the original join chain produced. Each branch's `child` is the lookup **wrapped in a `FilterNode` carrying the original equi-condition** — the runtime evaluates this filter over each lookup row with outer-side `ColumnReferenceNode`s resolving through `rctx.context` (the parent fork's snapshot already carries the outer slot, set by `runFanOutLookupJoin` before the fork).
7. Rebuilds the chain wrappers on top of the FanOut and reconstructs the surrounding Project with preserved attribute IDs.

## Use cases to validate

`test/optimizer/parallel-fanout.spec.ts` covers (10 tests, all passing):

- Below `minBranches` → no rewrite.
- At `minBranches`+ with cost-gate win (test fixture lowers `concurrency` to 2 in `beforeEach` to make N=3 surface a positive gate) → fan-out replaces the join chain.
- Local-only chains (memory vtab, latency=0) → no rewrite regardless of N.
- `concurrencyCap ≥ N` → no rewrite (formula clamps savings to 0).
- N=9 vs default `concurrency`=8 → rewrite fires (the "concurrency-bound" case).
- `disabledRules: Set(['fanout-lookup-join'])` honored.
- INNER branch with nullable FK → not eligible (one bad branch breaks the chain — the rule bails out conservatively).
- Execution equivalence with rule on vs. off (row counts + column values match).
- Output column names preserved across the rewrite (attribute-ID-preservation manifestation).
- `tuning.parallel.minBranches` override (raising it above N suppresses).

Test fixture is a `HighLatencyMemoryModule extends MemoryTableModule { expectedLatencyMs = 25 }` — declared in the spec and registered under `using hi_lat_memory`. No other changes are needed at the runtime level because the module's `expectedLatencyMs` propagates through `TableReferenceNode.computePhysical`.

Strict-fork compatibility verified: `QUEREUS_FORK_STRICT=1 mocha …/parallel-fanout.spec.ts` passes all 10.

## Honest gaps for the reviewer

1. **`expectedLatencyMs` is 0 in production by default.** Only the synthetic `HighLatencyMemoryModule` in the test spec declares a non-zero value. The rule is therefore inert until a remote-vtab plugin populates `VirtualTableModule.expectedLatencyMs`. This is by design (the ticket asks for it explicitly — "until a real plugin populates it, the cost gate stays inert by design, which is the desired v1 behavior"), but the next consumer who wants to actually exercise the rule end-to-end will need to add that plugin field somewhere real.

2. **Cost-gate formula has an opinionated quirk.** Per the ticket: `savings = (N − cap) × latency`. When `cap ≥ N`, savings clamps to 0 even though a 1-wave parallel fan-out *would* save against a serial nested loop (the nested-loop's wall-clock is N×latency, not 1×latency). The ticket explicitly specifies this formula, so the rule fires only when concurrency-bound. The "clusters when N > default concurrency cap" test (N=9, cap=8) pins this behavior. If the formula is ever revised, several tests will need to update — flag this if you think the simplification is hiding a perf win we should grab.

3. **Aggregate-rooted recognition is deferred.** Per the ticket's "Open questions", the aggregate variant (analogous to `ruleJoinEliminationUnderAggregate`) is deferred to a follow-up backlog ticket. Project-rooted recognition covers the motivating example.

4. **The rule bails out on *any* non-eligible branch in the chain.** If branches[middle] fails recognition (e.g., a residual non-equi predicate), the entire cluster is abandoned — the rule does not currently "skip the bad branch and cluster the rest". The single-branch fall-through to `join-elimination` (priority 24) handles trivial cases, but a chain like `[ok, ok, residual, ok, ok]` will produce no fan-out at all. Documented as conservative behavior; revisit if real-world chains demand partial clustering.

5. **Connection-per-branch acquisition is not implemented.** v1 always reuses the outer's connection. `'serial'`-mode branches serialize via the lock and get no concurrency win — correctness is preserved but the parallelism payoff is module-mode-gated.

6. **`FanOutBranchSpec` carries no per-branch equi-pair surface.** `FanOutLookupJoinNode.computePhysical` therefore still uses empty equi-pair lists in its `propagateJoinFds` calls, even though this rule has the equi-pair info. Tightening the FD propagation is a follow-up tracked on the node side, not blocking here.

7. **Adaptive concurrency and latency-driven branch ordering are out of scope** per the plan ticket.
