`Database.watch(scope, handler)` is now wired end-to-end. The watcher
is plan-independent: any `ChangeScope` value (analyzed, deserialized,
or hand-built literal) registers and fires correctly. The gate test
proving this (hand-built scope, no `Statement` involved) is in
`change-scope.spec.ts` and passes.

## What landed

### Public exports added (re-exported from `packages/quereus/src/index.ts`)

Types from `planner/analysis/change-scope.ts`:
- `Subscription`
- `WatchEvent`
- `MatchedWatch`
- `WatchHandler`

Method on `Database`:
- `watch(scope: ChangeScope, handler: WatchHandler): Subscription`

(Internal but exported for use by `Database.watch`):
- `subscriptionFromChangeScope` from `runtime/delta-executor.ts`
- `ChangeScopeTableInfo`, `SubscriptionFromChangeScopeContext`,
  `SubscriptionFromChangeScopeResult` types

### Architecture

`WatcherManager` (`core/database-watchers.ts`) owns the watcher half:

- Constructs its own `DeltaExecutor` (separate from the assertion
  evaluator's — assertions run pre-commit, watchers run post-commit).
- `watch(scope, handler)` validates, resolves column names to indices
  via `SchemaManager.findTable`, registers capture-spec demand for
  non-PK columns, builds the `DeltaSubscription` via
  `subscriptionFromChangeScope`, and returns a `Subscription` handle
  whose `unsubscribe` is idempotent and releases all resources.
- `runPostCommit()` is invoked from `TransactionManager.commitTransaction`
  after every connection commits and before the change log is cleared.
  Errors are caught and logged — watchers never roll the commit back.
- Listens for `SchemaChangeNotifier` `table_removed`/`table_modified`
  events and **disposes** affected subscriptions (no auto-rebind in v1
  — documented).

`subscriptionFromChangeScope` (`runtime/delta-executor.ts`) is the
pure shape-translation helper. It maps the public scope to a
`DeltaSubscription` per the table in the implement ticket:

| `WatchScope.kind` | Internal `BindingMode`             | Notes |
| ---               | ---                                | ---   |
| `full`            | `{kind: 'global'}`                 | Capture spec for non-PK named columns when `columns !== 'all'`. |
| `rows`            | `{kind: 'row', keyColumns}`        | Apply intersects kernel tuples against literal `values`. |
| `groups`          | `{kind: 'group', groupColumns}`    | Hits = kernel-emitted tuples. |
| `rowsByGroup`     | `{kind: 'group', groupColumns}`    | Apply intersects kernel tuples against literal `values`. |

Synthesized relation key per watch: `<schema>.<table>#watch_<i>`.

### Firing semantics implemented

- One `WatchEvent` per successful commit, with one `MatchedWatch` per
  touched watch.
- **Empty match → no fire.** A `rows` watch whose intersection is empty
  is *omitted* from `matched`; if every watch in the scope ends up
  empty, the handler is not called at all.
- Handler may be sync or async; rejection logged, swallowed.
- Re-entrancy is safe because `DeltaExecutor` already iterates a
  snapshot of subscriptions per run.

### txnId strategy

**Chosen:** a monotonic counter inside `WatcherManager`, minted at the
top of `runPostCommit()` and exposed to the subscription via
`SubscriptionFromChangeScopeContext.getCurrentTxnId()`. Format
`txn:<n>`. **Why:** no transaction-id surface exists on
`TransactionManager` today; the ticket explicitly allowed this
fallback ("opaque string from a monotonic counter inside `Database` if
no transaction-manager-assigned id exists"). Stable within one
`Database` instance, not portable across processes — documented.

### Schema-change invalidation timing

**Synchronous.** `WatcherManager` subscribes to
`SchemaManager.getChangeNotifier()`, which fires listeners
synchronously when DDL completes. So invalidating subscriptions for
the dropped/altered table happens inline with the DDL call. This is
the simplest correct behavior — a subscription whose table no longer
exists would otherwise be wired to non-existent capture specs.

### Capture-spec wiring

- `rows`, `groups`, `rowsByGroup` watches register the non-PK
  key/group columns for capture (mirrors the assertion path).
- `full` watches with `columns: 'all'` register nothing.
- `full` watches with a non-`'all'` column set register the non-PK
  named columns. **See "Known divergence" below**: the column-level
  narrowing on full+columns watches is best-effort; we register the
  spec but always fire on any change for v1.

### Validation

`watch(scope, handler)` rejects synchronously when:
- `scope.unboundParameters.length > 0` (error message mentions
  `bindParameters`).
- Any `TableWatch` references a missing table.
- Any column in `key` / `groupBy` / `columns` (when not `'all'`) does
  not exist.

Accepts and warns once (via the logger) when the scope is dead
(`watches.length === 0 && nonDeterministicSources.length === 0`).

## Known divergences from the design in the implement ticket

### Column tracking on `full` watches is best-effort (over-fires)

The ticket asked: "only fire if the captured changes touched at least
one of those columns" for `{kind: 'full', columns: {a, b}}` watches.

I register the capture spec for the named columns (as specified) so
the per-row column-level deltas are captured, **but the watcher fires
on every change** to that table regardless of whether a/b actually
changed. The reason: `TransactionManager.getChangedTuples` returns a
deduplicated tuple set across all rows in the txn, with no per-row
OLD/NEW distinction. There's no existing API to ask "did column X
change on this commit," and adding that surface (a per-row OLD vs NEW
projection comparison, or a new `hasColumnChanges` predicate on
`TransactionManager`) felt like out-of-scope infrastructure for the
v1 watcher ticket — particularly because the existing behaviour is
**sound** (never misses a fire), just coarse.

This is documented in `docs/change-scope.md` under "Known limitations
(v1)" and surfaced explicitly to the reviewer. If precise narrowing
matters, a follow-up should add the `TransactionManager` predicate.

### Global fallback on `rows`/`rowsByGroup`: surface every literal value

The ticket spec doesn't say what to do when the kernel's cost
fallback fires (or PK is missing) and a `rows` watch's binding gets
demoted to `global`. I chose to surface every literal value the watch
was registered for in `hits` — "all of your watched keys may have
changed" — rather than fire with empty hits or skip. Most informative
without over-firing on unrelated tables. Documented.

## Suggested review focus

- Has anyone wired `runPostCommitWatchers` into the right place in the
  commit path? It runs after `commitSucceeded = true`, inside the
  same outer try, before the outer-finally `clearChangeLog()`. The
  ticket explicitly required after-commit, before change-log clear.
  Worth double-checking against rollback semantics: on connection
  commit failure we throw before reaching the watcher call → correct.
- The plan-independent gate test: confirm the hand-built `ChangeScope`
  registers without any plan/statement involvement.
- The schema-change invalidation policy: dispose-on-change vs
  auto-rebind. I documented this prominently because it could surprise
  callers.
- The over-fire on `full + columns`: is this acceptable for v1 or
  should the precise narrowing land here? My read is "acceptable for
  v1, plumb the precise predicate later."
- The fuzz flake: this is pre-existing; the property-based
  `predicate-pushdown` differential test occasionally fails on
  randomly-generated SQL. Worth a separate fix-ticket if it's been
  flaking.

## What the reviewer should treat as a starting point

The watcher infrastructure is end-to-end correct for the common cases
(rows/groups/rowsByGroup/full with all columns) per the integration
tests. The two areas where my tests are a *floor* not a *ceiling*:

1. **Column narrowing on full+columns** — only "fires at all" is
   tested. Behavior when only an unwatched column changes is *not*
   asserted (because we currently over-fire).
2. **Schema-change invalidation** — disposing on `table_removed` and
   `table_modified` is implemented and the listener is wired; not
   covered by a dedicated test in this ticket. A targeted test
   (subscribe → drop table → mutate → assert handler did not fire,
   and that `Subscription.unsubscribe` is still safe) would tighten
   confidence here.

All other behaviour (intersection, plan-independence, unsubscribe
idempotency, multi-table, handler errors swallowed, validation) is
asserted directly.
