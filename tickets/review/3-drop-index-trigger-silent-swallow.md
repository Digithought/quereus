## Summary

Fixed `DROP INDEX` being silently ignored in the block planner. The `drop` case in `block.ts` only handled `table`, `view`, and `assertion` — `index` fell through with `break`, producing `undefined` which was filtered out at line 89.

This bug meant `DROP INDEX IF EXISTS` statements generated by the declarative schema system (`schema-differ.ts:262`) during `APPLY SCHEMA` migrations were silently swallowed, potentially leaving orphaned indexes.

## What was built

### DROP INDEX full pipeline
- **DropIndexNode** (`planner/nodes/drop-index-node.ts`): Plan node following DropViewNode pattern
- **buildDropIndexStmt** (`planner/building/drop-index.ts`): Builder extracting schema/index name from AST
- **emitDropIndex** (`runtime/emit/drop-index.ts`): Emitter delegating to `SchemaManager.dropIndex()`
- **SchemaManager.dropIndex()** (`schema/manager.ts`): Searches all tables in target schema to find the owning table, calls `module.dropIndex()`, removes index from table schema, emits change notifications and schema events
- **VirtualTableModule.dropIndex()** (`vtab/module.ts`): New optional method on module interface
- **MemoryTableModule.dropIndex()** (`vtab/memory/module.ts`): Implementation delegating to LayerManager

### Block planner routing
- `block.ts`: Added `'index'` case routing to `buildDropIndexStmt`
- `block.ts`: Added defensive `'trigger'` case that throws UNSUPPORTED (parser currently rejects `DROP TRIGGER` before planner is reached, but this guards against future parser changes)

### Emitter registration
- `register.ts`: Wired `PlanNodeType.DropIndex` → `emitDropIndex`
