## What was built

Fixed `DROP INDEX` being silently ignored in the block planner. The `drop` case in `block.ts` only handled table, view, and assertion — `index` fell through with `break`, producing `undefined` which was filtered out. This meant `DROP INDEX IF EXISTS` statements generated by the declarative schema system during `APPLY SCHEMA` migrations were silently swallowed, potentially leaving orphaned indexes.

### Full DROP INDEX pipeline
- **DropIndexNode**: Plan node following DropViewNode pattern
- **buildDropIndexStmt**: Builder extracting schema/index name from AST
- **emitDropIndex**: Emitter delegating to SchemaManager.dropIndex()
- **SchemaManager.dropIndex()**: Searches tables in target schema to find owning table, calls module.dropIndex(), removes index from table schema, emits change notifications and schema events
- **VirtualTableModule.dropIndex()**: Optional method on module interface
- **MemoryTableModule.dropIndex()**: Implementation delegating to LayerManager

### Defensive DROP TRIGGER guard
Added explicit `'trigger'` case in block.ts that throws UNSUPPORTED, preventing future silent swallow if the parser is extended.
