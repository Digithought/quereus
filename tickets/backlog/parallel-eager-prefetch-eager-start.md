description: Make EagerPrefetch start its pump on emit (run()) instead of on first iteration, so a hash join's build phase overlaps with the probe's first prefetch round-trip.
files: packages/quereus/src/runtime/emit/eager-prefetch.ts, packages/quereus/src/runtime/emit/bloom-join.ts, packages/quereus/src/planner/nodes/eager-prefetch-node.ts
----

`EagerPrefetchNode` is meant to hide latency by pumping its source into a bounded ring buffer ahead of the consumer's first demand. Today the pump is deferred: `emitEagerPrefetch` returns an async generator whose body (the `prefetchAsyncIterable` fork + pump) only runs on the first `.next()`. In a `BloomJoinNode`, the runtime drains the build (`right`) side to completion *before* it ever touches the probe (`left`) side, so when the probe is an `EagerPrefetchNode` its pump does not start until after the entire build phase has finished. The immediate win the `eager-prefetch-probe` wrap rule already delivers is per-row probe-latency hiding (the buffer fetches the next batch while the join does its synchronous probe work), but the larger "build phase overlaps with the probe's first round-trip" win is left on the table because the pump starts too late.

**Concurrency-safety gate (do not skip).** Today the `eager-prefetch-probe` rule has *no* concurrency-safety gate, and it is correct not to: the BloomJoin drains the build (`right`) side to completion before the probe pump ever starts, so the pump and the build never run at the same time, and the pump is the sole reader of the probe iterator. The moment this ticket makes the pump start on `run()`, that invariant breaks — the probe pump will iterate the `left` subtree *concurrently* with the build for-await over the `right` subtree. If both sides sit over the same `'serial'` vtab connection (or otherwise share a non-reentrant cursor), concurrent iteration corrupts state. So this change must be paired with a `concurrencySafe` gate on the wrap rule (mirroring `rule-async-gather-union-all`, which already gates on `child.physical.concurrencySafe`): only wrap the probe in EagerPrefetch when both `probe.physical.concurrencySafe` and `build.physical.concurrencySafe` are true. Land the gate in `rule-eager-prefetch-probe.ts` as part of this ticket, with a test that a serial/non-safe probe is left unwrapped.

The goal is to start the prefetch pump as soon as the scheduler invokes the prefetch's `run()` — which happens before the BloomJoin's `run()` drives the build for-await — so the probe's first network round-trip runs concurrently with the build materialization. Two viable approaches: (1) restructure `emitEagerPrefetch` so `run()` returns a non-generator `AsyncIterable` whose construction has already kicked off the fork+pump (no laziness from the generator body); or (2) modify `emitBloomJoin` to obtain `leftSource[Symbol.asyncIterator]()` and pull one `.next()` before the build for-await, triggering the prefetch generator body early. Approach (1) is cleaner and benefits all consumers, not just hash joins; approach (2) is more surgical but couples the join emitter to prefetch internals. Either must respect the strict-fork contract (the gather rule documents a known Sort-above-fork interaction) and preserve current row/order/attribute semantics. Validate with a latency-instrumented runtime test (or timing assertion) showing the build wait and the probe's first fetch overlap; the existing `parallel-eager-prefetch-probe.spec.ts` execution-equivalence test guards correctness.
